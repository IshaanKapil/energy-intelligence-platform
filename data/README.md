# Data

Hourly energy dataset for the Energy Intelligence Platform.

## Files

| File | What it is |
|---|---|
| `energy_dataset.csv` | **Main file** — everything merged on `datetime` (use this) |
| `load.csv` | Industrial electricity demand only |
| `solar_weather.csv` | Weather + solar generation only |
| `price.csv` | Electricity price only |
| `sample_week.png` | Visual sanity check of one week |

## Columns (`energy_dataset.csv`)

| Column | Unit | Meaning |
|---|---|---|
| `datetime` | timestamp | Hourly index |
| `load_mw` | MW | Industrial electricity demand (**target for load forecasting**) |
| `temperature_c` | °C | Ambient temperature |
| `cloud_cover_pct` | % | Cloud cover 0–100 |
| `irradiance_wm2` | W/m² | Solar irradiance (GHI) |
| `solar_mw` | MW | Solar generation, 20 MW plant (**target for solar forecasting**) |
| `price` | /MWh | Electricity price (**target for price forecasting**) |

## Synthetic vs. Real

- `energy_dataset.csv` is **synthetic but realistic** — generated by
  `generate_synthetic_data.py` with transparent relationships (hour-of-day
  shift pattern, weekday/weekend, temperature U-curve, a slow production
  random walk, autocorrelated noise; solar from a solar-geometry curve
  attenuated by cloud cover; price tracking demand + peak hours). This means
  SHAP will surface *sensible* drivers, and you can explain exactly how every
  signal was produced.
- For a stronger placement story, run `fetch_real_data.py` on your own
  machine to pull **real** PJM load + real Open-Meteo weather/solar into
  `energy_dataset_real.csv`. Same column names, so your models don't change —
  just point them at the real file.

## Verified relationships (from synthetic data)

- solar ↔ irradiance correlation ≈ 1.0
- price ↔ load correlation ≈ 0.7 (positive)
- weekday avg load ≈ 48 MW vs weekend ≈ 39 MW
- solar is exactly 0 at night
